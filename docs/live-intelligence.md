# Live intelligence — the in-meeting brief

Status: **proposed** (spec). Extends DESIGN.md §8 (Summarization), which covers
the *end-of-meeting* summary. This covers the *during-meeting* companion: a
sidebar a small local model keeps filled out in real time — next steps, open
questions, suggested questions, and drafted answers to questions asked of you.

## What this is

While a meeting runs, a **`LiveAnalyst`** watches the finalized live transcript
and maintains a rolling **brief** with a few structured sections:

- **Next steps** — action items as they're agreed, with owner and timestamp.
- **Open questions** — raised but not yet answered; resolved ones get marked,
  not deleted.
- **Suggested questions** — things the model thinks *you* might want to ask.
- **Asked of you** — questions directed at you off the system track, each with a
  drafted answer grounded in the conversation so far.

Everything runs on-device (Apple Foundation Models on the ANE). Nothing here
touches audio — the analyst consumes finalized transcript *text* only.

## Why a fold, not a summary

Apple FM has a hard **4096-token context** (instructions + prompt + output —
DESIGN.md §8.1), so "re-summarize the whole meeting every 30 seconds" is
impossible past the first few minutes, and map-reduce is far too slow to repeat
live. Instead the analyst is a **fold**:

```
brief₀ = empty
briefₙ = FM( instructions + briefₙ₋₁ + transcript delta since last pass )
```

The prompt is bounded no matter how long the meeting runs: the previous brief is
compact (it's the distillate), and the delta is capped by the trigger cadence.
Items naturally persist across passes, get amended, or get resolved ("open
question → answered at 14:32") because the model always sees the prior state.
This is also what makes the UI *stable* — the model updates a document rather
than regenerating one.

## Inputs

- **Finalized live segments only.** The analyst taps the same commit point that
  appends to `TranscriptLog` (`transcript.md`) — the moment a live segment goes
  volatile→final in `CaptureModel`. Volatile text is never fed to the model; it
  mutates, and re-analyzing text that later changes wastes tokens and produces
  flicker.
- **Speaker attribution** comes with each segment (live identity: enrolled
  names, live-named speakers, or "Speaker N"). The brief attributes items to
  people — "Alex asked…", "Tom owns…" — which is why this feature sits *after*
  live diarization in the build order: a brief that attributes action items to
  the wrong person is worse than no brief.
- **Track separation is load-bearing** for the "asked of you" section: the mic
  track is always you, so a question arriving on the *system* track followed by
  speech on the *mic* track is, for free, "a question addressed to me." No
  mixed-stream app has this signal.
- *(Later)* the calendar attendee roster (DESIGN.md §6) and cross-meeting search
  give the answer-drafting step richer grounding ("you discussed this June 12").

## The brief schema

`@Generable` guided generation (constrained decoding — output always parses,
even from a 3B model):

```swift
@Generable
struct LiveBrief {
    @Generable
    struct Item {
        var text: String
        var speaker: String?     // who said / owns it
        var resolved: Bool       // answered / done — kept, shown struck-through
    }
    var nextSteps: [Item]
    var openQuestions: [Item]
    var suggestedQuestions: [Item]
}
```

The schema itself costs context tokens (§8.1's lesson), so it stays lean; the
"asked of you" section is deliberately **not** in this schema — it's a separate,
focused call (below), not part of the fold.

Timestamps are *not* generated by the model (it would hallucinate them). The
diff layer stamps each new/changed item with the time of the pass that produced
it, and tap-to-seek uses that.

## Cadence

Per-segment is too chatty; a fixed timer fires mid-thought. Trigger on **segment
finalization**, gated:

- **Floor:** don't run until ~30–45 s of new finalized speech (word count, not
  wall clock) has accumulated since the last pass.
- **Prefer turn boundaries:** once past the floor, fire on the next speaker
  change or sentence-final segment rather than mid-monologue.
- **One in flight:** a single serialized generation task; segments arriving
  during a pass coalesce into the next delta. If generation is slower than the
  cadence, the cadence stretches — never a queue.
- **Meeting end:** one final fold pass over the tail, then the brief freezes.

## "Asked of you" — two stages, not part of the fold

1. **Detector (cheap, per-segment):** runs on every finalized *system-track*
   segment. Heuristics first — question-shaped sentence (interrogative form /
   trailing `?` from the ASR), optionally containing your name, not
   self-answered by the same speaker's next segment. No LLM required to detect.
2. **Answer draft (focused FM call):** on detection, a single-purpose prompt —
   the question + the last N turns of context — drafts a 1–2 sentence suggested
   answer. This is a separate small call precisely so the fold's prompt stays
   lean and the answer arrives in seconds, not at the next fold pass.

Drafts are ephemeral: shown, then auto-dismissed once you've answered (speech on
the mic track following the question) or after a timeout. They're assistance,
not transcript.

## UI stability rules

Items that flicker, reword, or reorder every pass destroy trust. The view model
enforces:

- **Diff, don't replace.** New brief is diffed against the old by fuzzy item
  match; unchanged items keep their identity, position, and timestamp. Only
  genuine adds/edits/resolutions animate.
- **The prompt says so too:** instructions tell the model to carry prior items
  forward verbatim unless new transcript contradicts or resolves them.
- **Pin / dismiss per item.** Pinned items are never dropped by the model
  (re-asserted mechanically after each pass). Dismissed items go into a
  suppression list appended to the instructions ("do not resurface: …").
- **Resolved ≠ deleted.** Resolved items render struck-through and collapse,
  preserving the meeting's shape.
- Every item is **tap-to-seek** into the transcript (timestamp from the diff
  layer; the word-level seek machinery already exists).

## Engine & placement

Apple FM is the engine: zero setup, guided generation, and it runs on the
**ANE** — consistent with the "live ML stays off the GPU" invariant. MLX would
give a smarter brief but contends for the GPU and takes seconds per pass; if
ever wanted, it slots in behind the same protocol as a user choice (mirroring
the `Summarizer` split). One open question to measure: ANE contention between
bursty FM passes and Sortformer's continuous ~480 ms updates — expected fine
(FM passes are seconds apart), but verify live latency doesn't degrade during a
pass.

The analyst must obey the durability invariant: it's fully downstream of
capture, failures are logged and skipped (the next pass folds a bigger delta),
and a crash in FM can never touch the recording path.

## Decisions

- **D1 — engine.** Apple FM (recommended: ANE, free, guided generation) vs MLX
  live (quality, but GPU + latency). Behind a protocol either way.
- **D2 — protocol shape.** *Not* `Summarizer` — that contract is one-shot
  `transcript → MeetingNotes`. The analyst is a stateful fold with its own I/O
  (`(LiveBrief, delta) → LiveBrief`). New small protocol (`LiveAnalyst`) in
  `Summarization/`, sharing prompt utilities with the summarizers.
- **D3 — fold state representation.** Feed the previous brief back as compact
  rendered text (recommended: cheapest tokens, most natural for the model) vs
  JSON (exact, but verbose and brittle at 4k context).
- **D4 — question detector.** Heuristics (recommended first: free, latency
  ~zero) vs a tiny FM classification call per candidate (better recall on
  indirect questions — "I'd love Eric's take"; add only if heuristics miss too
  much in practice).
- **D5 — relationship to the final summary.** The brief is provisional, like
  the live transcript: at meeting end it's stored as a snapshot (useful eval
  artifact + instant notes while the final pass runs), but the authoritative
  `MeetingNotes` come from the Phase-5 `Summarizer` over the final transcript.
  The brief may be offered to the final summarizer as a hint, never as ground
  truth.

## Build phases

- **Phase L1 — replay spike (no UI).** A `saywhat` CLI subcommand that replays
  an existing recording's finalized segments as if live and prints each fold
  pass's brief. This is where prompt, schema, and cadence get tuned cheaply,
  against real meetings, with reproducible input. Exit criterion: a brief you'd
  actually trust from one of your recorded calls.
- **Phase L2 — `LiveAnalyst` core.** The fold actor: delta accumulation,
  cadence gate, serialized passes, diff, pin/dismiss/suppression. Generation
  goes behind the protocol so unit tests script it (deterministic fake engine)
  — same pattern as the diarization tests, no CoreML in tests.
- **Phase L3 — sidebar UI.** Sections, diff-driven animation, tap-to-seek,
  pin/dismiss. Brief snapshot persisted with the session.
- **Phase L4 — "asked of you."** System-track detector + focused answer call +
  ephemeral card UI.
- **Phase L5 — grounding upgrades (later).** Calendar roster in the prompt;
  cross-meeting retrieval for answer drafts.

## Test plan

- **Fold unit tests (scripted engine, no models):** delta accumulation and
  cadence floor; coalescing while a pass is in flight; diff keeps unchanged
  items' identity/timestamps; pinned items survive a model that drops them;
  dismissed items never resurface; resolved items are kept, not deleted.
- **Detector unit tests:** question on system track → candidate; mic-track
  question → not "asked of you"; self-answered question suppressed.
- **Golden replay fixture:** run L1 replay over a benchmark recording, snapshot
  the brief's evolution; review by rubric (are the action items real? owners
  right? junk rate?) rather than exact-match — LLM output isn't
  byte-reproducible even at low temperature.
- **Non-interference:** capture + live transcript latency measured with the
  analyst on vs off; a thrown/hung FM pass (watchdog, like `FinalPass`) never
  affects recording.
